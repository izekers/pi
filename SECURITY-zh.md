# 安全策略

本文档旨在帮助你理解 Pi 背后的安全概念及其边界所在。

总体而言，Pi 是一个在本地运行 coding agent，其运行于启动它的用户的安全边界内。用户有责任监控其操作，或将其限制在容器、虚拟机或其他沙箱解决方案中。

Pi 将本地用户账户及该账户可写的文件视为与 Pi 进程处于同一信任边界内。如果攻击者能够修改用户主目录、工作区、shell 启动文件、环境变量或 Pi 配置下的文件，他们通常可以影响 Pi 或其他本地开发工具。除非报告能够证明 Pi 本身授予了某种写权限或跨越了操作系统特权边界，否则依赖此类事前写权限的报告不构成安全漏洞。

Pi 依赖于用户安装可信的扩展程序、加载可信的技能，并且只在可信的仓库中使用 Pi。这是因为 `AGENTS.md` 或注释中的指令可以被轻易用于对 coding agent 进行提示注入，且无法对此加以保护。

## 报告漏洞

如果你发现 Pi 或本仓库中其他软件包存在安全漏洞，请通过以下方式私下报告：

- 发送邮件至 `security@earendil.com`，或
- 通过 GitHub 安全咨询（GitHub Security Advisories）提交私密报告

请在报告中包含：

- 问题描述及其影响
- 复现步骤、概念验证（PoC）或相关日志
- 受影响的软件包、版本、提交（commit）或配置
- 任何已知的缓解措施

请勿为安全相关报告开启公开 issue。我们将审核报告并在适当时机协调披露。

## 范围

以下内容在范围内：分发的软件包、命令行工具、API 和仓库代码，以及 `pi.dev` 上由 Earendil 运营的基础设施。

## 范围之外

- 本地代码执行或沙箱行为（Pi coding agent 有意不提供沙箱）
- 用户安装的 Pi 扩展或技能的行为
- 在不可信仓库中工作的风险
- 安装不可信的扩展、技能、软件包或工具的风险
- 不可信的 MITM 代理导致的问题
- Pi 安装实例对公共互联网的暴露
- 提示注入攻击
- 属于第三方或用户控制的凭证所暴露的密钥
- 需要具备在目标机器上创建、修改、删除或替换文件、目录、符号链接、环境变量、shell 配置或其他用户控制的本地状态的能力的报告。包括 `~/.pi`、`~/.pi/agent/models.json`、工作区文件、`AGENTS.md`、技能、扩展、扩展配置、dotfiles 以及通过 NFS、漫游配置文件或 dotfile 管理器同步的文件——除非报告能证明 Pi 本身授予了这种访问权限。
- 因刻意削弱用户配置而导致的问题。
- 需要对 Pi coding agent 提供可信本地输入/配置才能成立资源消耗/拒绝服务（DOS）类报告。
- 关于恶意模型输出的报告。
- 用户已批准或用户发起的本地操作被作为漏洞报告的情况。

## 给报告者的说明

最有价值的报告能够展示当前可复现的安全边界绕过并附有明确的影响演示。仅展示预期的本地 agent 行为、提示注入或恶意可信扩展/技能的报告，在本模型下不构成安全漏洞。

例如，一份报告显示写入可信 Pi 配置文件中的恶意内容会导致 Pi 执行命令、加载攻击者控制的工具、将凭证发送到攻击者控制的端点或以其他方式改变行为，这属于范围之外。

请尽可能提供受影响的确切路径、软件包版本或提交 SHA、配置以及针对最新版本或最新 `main` 分支的概念验证。对于依赖项报告，需提供证据表明分发的依赖项受到影响，且该问题可通过 Pi 触及。对于暴露密钥的报告，需提供证据证明该凭证归 Earendil 所有或可访问由 Earendil 运营的基础设施或服务。
